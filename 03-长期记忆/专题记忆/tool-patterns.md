# 工具调用规范

## 复杂任务前自检
- 任务≥5步工具调用前，先读 stability-status.md 确认系统状态
- 网关 stopped 时避免调用依赖网关的工具
- 连续失败≥2时主动切换策略而非无限重试

## 反馈闭环（SmartControl 控制论启发）
- 连续 2 次同工具失败 → 换不同方法重试，而非重复相同调用
- 连续 3 次不同工具仍失败 → 退回任务重规划，请求用户输入
- 每次失败记录到上下文供后续决策

## 工具输出校验（Cyber4OT 安全验证启发）
- 工具返回异常结果（空数据、错误码、越界值、不合理占比）时，验证后再动作
- 验证方式：重新读取源数据 / 交叉检查 / 向用户确认
- 经验法则：工具输出是"原始证据"，不是"最终结论"

## 调用日志链（pyjevsim 日志回放启发）
- 单次任务中维护隐式调用序列（不写文件，上下文跟踪即可）
- 记录：工具名 → 参数摘要 → 结果状态(成功/失败/异常) → 策略是否切换
- 作用：失败时回溯调用链，而不是孤立地看最后一次失败

## 输出自检（auto-xFS 特征选择启发）
- ≥3步工具调用的任务，输出前做快速质量自评三问：
  ① 结果完整吗？有没有遗漏的维度？
  ② 逻辑自洽吗？工具输出之间有无矛盾？
  ③ 哪些点我不确定？需标注置信度或请用户验证
- ≤3步的简单任务跳过自检

## 代码自审（PuppyCodeReview AI代码审查启发）
- 生成代码片段后交付前自查三项：
  ① 空指针/越界：列表索引、dict key、None 检查
  ② 类型一致：传入值和函数签名匹配
  ③ 边界条件：空输入、超大输入、特殊字符
- 不追求完美，但要过滤明显 Bug

## 技能定义规范 (AIDDL 领域定义语言启发)
- 每个技能必须包含：name, description, tags, trigger 条件, numbered steps, pitfalls, verification
- 技能输出应可被其他技能消费（标准化接口）

## 工具集成契约（eidos 外部函数集成启发）
- 任何新工具的集成三步法：
  ① 定义输入参数（类型 + 约束 + 默认值）
  ② 定义成功输出结构（字段 + 类型 + 含义）
  ③ 定义错误输出结构（错误码 + 错误信息 + 恢复建议）
- 工具不应产生副效应（纯函数化）
