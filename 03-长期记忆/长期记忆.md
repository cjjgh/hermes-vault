以下为条件触发规则，每次启动自动生效。规则编号用于审计追溯。

【规则1：先计划再执行】
WHEN 第1次工具调用前（非累计2次后）且任务需≥2步
→ THEN 输出计划：{variables:[], steps:[], expected:{tool:预期值}}。expected 禁止空值——代码填目标函数名，数值填范围或精确值。计划存到当前会话 checkpoint（每次更新覆盖前值），供后续规则读取。

【规则2：同类工具批处理】
WHEN 准备按顺序第3次调用（不计重复性质的失败重试）
→ THEN 暂停，检查分类表（read/write/execute/search/web），同类工具合并为单次批量执行

【规则3：预期结果偏差处理】
WHEN 某个工具调用返回码/空结果与【规则1】checkpoint 中对应预期值不符（checkpoint 不存在或为空则跳过此规则）
→ THEN 检查当前方法累计尝试次数——连续3次结果无实质变化才切换策略，否则继续收敛

【规则4：可验证输出的外部测试】
WHEN 工具返回的结果声明为"已完成"或"交付物"
→ THEN 按类型执行外部测试：代码→pytest/pylint，命令→独立shell执行，数学→预期值匹配，配置→schema验证。测试在独立进程运行。测试失败意味着当前任务未完成。

【规则5：复杂任务五步闭环】
WHEN 工具调用累计达5次（按任务计数器计，计数器定义：收到新用户意图=重置为0，每个工具调用+1，不含子agent内部调用）
AND rule5_fired_this_task=false（防止嵌套自触发：规则5执行体中的调用不再次触发规则5）
→ THEN 加载并执行 reflexion-post-task 技能的完整流程（Step 0→4），不再内联重复步骤内容。
完成后 reflexion.log 追加结构化记录。

【规则6：代码审查P0/P1/P2】
WHEN 执行代码审查
→ THEN P0硬编码列表：SQL注入/硬编码密钥/命令注入/path traversal/data exfiltration。P1：空指针/越界/逻辑短路。精确到file:line。P0/P1触发 auto-fix 循环（最多2次），非阻断。

【规则7：交付前独立路径验证】
WHEN 交付最终结论前（未经【规则5】子agent审查的任务）
→ THEN delegate_task 起子 agent 用相反方法论推导（如我做了归纳推理→子 agent 做演绎验证），对比结论一致性
