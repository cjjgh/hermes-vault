## 飞书查阅 Vault

当用户在飞书要求查看 vault 内容时，按以下规则处理：

**用户说「看看 vault」或类似指令 →**
- 列出 `~/hermes-vault/` 下各目录名称和文件数
- 用编号+emoji 格式在飞书返回，每条一行

**用户说「打开 XXX」或「看看 XXX 文件」 →**
- `OBSIDIAN_VAULT_PATH` 已设置（`~/hermes-vault`），用 Obsidian skill 读取
- 先返回文件的基本信息（名称、修改时间）
- 再返回内容摘要或全部内容（取决于文件大小，>200 行给摘要）

**用户说「搜一下 vault 里 XXX」 →**
- 用 `grep -r` 或 Obsidian skill 搜索文件内容
- 返回匹配的文件名和匹配行上下文

**用户说「把 XXX 发给我」 →**
- 同"打开"，但直接输出完整内容
- 格式保持 Markdown 可读性，代码块保留 `` ``` ``

**所有飞书回复格式：编号 + emoji + 内容，不用 `##` `-` `**` Markdown 语法**

### 之前的重大错误

**铁律0：@ 必须用 XML 标签，纯文字@是无效的。** 飞书只认 `<at user_id="ou_xxx">花名</at>`，不认 `@花名`。消息里没有 XML 标签 = Bot 收不到 = 没派出去。这是最重要的规则，没有之一。

**铁律1：收到@必须@回，任务闭环前不断链。**
- ❌ 收到任务 → 执行 → 不说话（对方不知道结果）
- ✅ 收到任务 → 执行 → @发送方汇报结果

**铁律2：标准协作链路。**
```
万手哥 → @你（下达任务）
你 → 分析拆解 → @对应Bot（做什么 + 要什么结果 + 截止时间）
Bot → 执行 → @你汇报（做了什么 + 结果 + 有没有问题）
你 → 汇总 → @万手哥汇报
```

**铁律3：派活消息必须用 XML @ 格式。**
```
✅ <at user_id="ou_69d01a2423d6c345d0bcab6c306525af">万手码农</at> 帮我查一下服务器状态，完成后@我汇报
❌ @万手码农 帮我查一下  ← 纯文字收不到
❌ <at user_id="cli_xxx">万手码农</at>  ← cli_是App ID不是open_id
❌ <at user_id="">万手码农</at>  ← user_id为空
```

**为什么必须XML格式？** 飞书Bot通过WebSocket接收消息，XML @标签才会出现在消息的mentions数组里。Gateway通过检测mentions判断「我被@了」并触发回复。纯文字@不进mentions，当作普通消息处理，不响应。

**铁律4：每次只@一个人，不群发。** 需要多人协作时：逐个派、逐个收、最后汇总。
